## 1. 개요

### 1.1 YM TestCase란?

YM TestCase는 테스트 케이스(TC)의 작성, 관리, 테스트 수행, 결과 분석 및 리포트 생성을 위한 **웹 기반 테스트 관리 시스템**입니다.

기존 엑셀 기반 TC 관리의 한계(버전 충돌, 협업 어려움, 이력 추적 불가)를 해결하고, 팀 단위 테스트 관리 프로세스를 표준화하기 위해 도입되었습니다.

### 1.2 도입 배경 및 목적

| 기존 방식 (엑셀) | YM TestCase |
|---|---|
| 파일 공유 시 버전 충돌 | 웹 기반 실시간 협업 |
| 테스트 결과를 수동으로 취합 | 자동 집계 및 대시보드 |
| 변경 이력 추적 불가 | 필드 단위 변경 이력 자동 기록 |
| 리포트 작성에 별도 시간 소요 | PDF/Excel 리포트 자동 생성 |
| 권한 관리 없음 | 역할 기반 접근 제어 (RBAC) |
| 첨부파일 관리 어려움 | TC별 첨부파일 업로드/미리보기 |

### 1.3 주요 기능 요약

| 기능 | 설명 |
|---|---|
| **프로젝트 관리** | 프로젝트 단위 TC/런 관리, 멤버 관리, 공개/비공개 설정 |
| **TC 관리** | TC CRUD, Excel Import/Export, 소프트 삭제/복원, 변경 이력 |
| **테스트 수행** | 런 생성/복제/완료, 결과 입력(키보드 단축키), 첨부파일, 타이머 |
| **비교 분석** | 두 런 간 결과 비교, 리그레션/개선 자동 감지 |
| **대시보드** | 요약 카드, 도넛/바 차트, 히트맵, 우선순위/카테고리/담당자별 분석 |
| **리포트** | 결과 요약 조회, PDF/Excel 다운로드 |
| **관리자** | 사용자 관리, 역할 변경 |

## 2. 접속 정보

### 2.1 시스템 접속

> **Info**: 프로덕션 URL은 인프라팀에서 별도 공지합니다.

### 2.2 계정 발급 절차

1. YM TestCase 로그인 페이지에서 **"회원가입"** 링크를 클릭합니다.
2. 아이디, 표시명, 비밀번호를 입력하여 가입합니다.
3. 가입 직후에는 **User** (일반 사용자) 역할이 부여됩니다.
4. **관리자(Admin)에게 역할 변경을 요청**합니다.
   - QA 엔지니어 → **User** (프로젝트 Tester 역할 부여)
   - PM / QA 리드 → **QA Manager** 역할 요청
   - 관리자 → **Admin** 역할 요청

> **Warning**: 최초 가입 사용자는 자동으로 Admin 역할이 부여됩니다. 시스템 최초 셋업 시에만 해당됩니다.

> **비밀번호 규칙**: 비밀번호는 **최소 8자** 이상이어야 합니다. 관리자가 비밀번호를 초기화한 경우 다음 로그인 시 강제 변경됩니다.

## 3. 업무 프로세스

### 3.1 전체 업무 흐름

```
프로젝트 생성 → TC 작성/Import → 테스트 런 생성 → 테스트 수행 → 결과 분석 → 리포트 공유
     (Admin)    (프로젝트 Admin)  (프로젝트 Admin)  (Tester+)      (전체)        (전체)
```

### 3.2 단계별 상세

#### STEP 1. 프로젝트 생성 (Admin)

1. 프로젝트 목록 페이지에서 **"+ 새 프로젝트"** 클릭
2. 프로젝트명, 설명 입력 후 생성
3. 설정 탭에서 Jira Base URL, 공개/비공개 설정
4. 멤버 추가 및 프로젝트 역할 부여

**프로젝트 네이밍 규칙 (권장)**:
- `[서비스명] [배포버전] [테스트유형]`
- 예: `모바일 앱 v7.3 정기배포`, `웹 결제 개선 스프린트25`

#### STEP 2. TC 작성 (프로젝트 Admin)

**방법 A: 직접 작성**
1. TC 관리 탭에서 **"+ 행 추가"** 클릭
2. 그리드에서 인라인 편집으로 TC 정보 입력
3. **"저장"** 클릭

**방법 B: 엑셀 Import (권장)**
1. 기존 엑셀 TC 파일 준비 (.xlsx, .xls)
2. **"Excel Import"** 클릭 후 파일 선택
3. 자동 헤더 매핑으로 TC 일괄 등록
4. 결과 확인 후 **"저장"**

> **Tip**: 50개 이상의 한글/영문 헤더 변형을 자동 인식합니다. 기존 엑셀 양식 그대로 가져올 수 있습니다.

**TC 작성 가이드라인**:

| 필드 | 작성 기준 | 예시 |
|---|---|---|
| TC ID | `TC-[모듈약어]-[번호]` 형식 | TC-PM-001 |
| Type | Func. / UI/UX / Perf. / Security / API / Data | Func. |
| Category | 대분류 (기능 영역) | 프로젝트 관리 진입 |
| Depth 1 | 중분류 (화면/모듈) | 엔터프라이즈 |
| Depth 2 | 소분류 (세부 기능) | 타이틀 |
| Priority | 매우 높음 / 높음 / 보통 / 낮음 / 매우 낮음 | 높음 |
| Platform | WEB / Mobile / iOS / Android / API / 공통 | WEB |
| Precondition | 테스트 전 필요한 조건 | 관리자 계정으로 로그인 |
| Test Steps | 번호 매기기, 한 단계씩 명확하게 | 1. 메뉴를 클릭한다. 2. ... |
| Expected Result | 각 단계별 기대 결과 | 1. 페이지가 표시된다. |

#### STEP 3. 테스트 런 생성 (프로젝트 Admin)

1. 테스트 수행 탭에서 **"+ New Test Run"** 클릭
2. 런 정보 입력:
   - **런 이름**: `[서비스명] [날짜] [라운드]` (예: `모바일 앱 20260316 R1`)
   - **라운드**: R1, R2, R3... (회차)
   - **버전**: 테스트 대상 버전 (예: v7.3.1)
   - **환경**: 테스트 환경 (예: STG, PRD)
3. 생성 시 모든 TC가 **NS(미실행)** 상태로 포함됨

#### STEP 4. 테스트 수행 (Tester)

1. 좌측 패널에서 테스트 런 선택
2. 결과 그리드에서 각 TC의 결과를 입력:

| 결과 | 의미 | 단축키 | 색상 |
|---|---|:---:|---|
| **PASS** | 기대 결과와 일치 | `P` | 녹색 |
| **FAIL** | 기대 결과와 불일치 (버그) | `F` | 빨간색 |
| **BLOCK** | 선행 조건 미충족으로 실행 불가 | `B` | 주황색 |
| **N/A** | 해당 없음 (테스트 범위 외) | - | 회색 |
| **NS** | 미실행 | `N` | - |

3. FAIL 시 **Actual Result** 필드에 실제 동작을 기록
4. 필요 시 **Issue Link**에 Jira 이슈 키 입력 (예: PROJ-123)
5. 증빙이 필요한 경우 **첨부파일 업로드** (스크린샷 등)

**효율적인 수행 팁**:
- **키보드 단축키** 활용: P/F/B/N 키로 빠른 결과 입력
- **Ctrl+D**: 현재 값을 아래로 복사 (동일 결과 반복 시)
- **Shift+Click**: 범위 선택 후 일괄 적용
- **Ctrl+Z**: 실수 시 즉시 실행 취소 (최대 200단계)
- **타이머 기능**: 각 TC별 수행 시간 기록 가능

#### STEP 5. 결과 분석 (전체)

**대시보드 활용**:
1. 대시보드 탭에서 테스트 런 선택
2. 확인 가능한 분석 항목:

| 분석 항목 | 활용 |
|---|---|
| 요약 카드 | 전체 진행률, Pass Rate 한눈에 파악 |
| 결과 분포 (도넛) | PASS/FAIL/BLOCK/N/A/미수행 비율 시각화 |
| 라운드별 비교 (바) | R1→R2→R3 추이 확인, 개선 여부 판단 |
| 우선순위별 분포 | 높은 우선순위 TC의 FAIL 건수 집중 확인 |
| 카테고리별 분포 | 어떤 기능 영역에 이슈가 집중되는지 파악 |
| 담당자별 현황 | 팀원별 진행률 및 완료율 확인 |
| 실패 히트맵 | 카테고리 × 우선순위 매트릭스에서 리스크 영역 식별 |

**비교 기능 활용**:
1. 비교 탭에서 이전 런(R1)과 현재 런(R2) 선택
2. **리그레션 필터**: PASS→FAIL로 악화된 항목만 조회
3. **개선 필터**: FAIL→PASS로 해결된 항목 확인

#### STEP 6. 리포트 공유 (전체)

1. 리포트 탭에서 테스트 런 선택
2. 결과 요약 확인 (총 TC, Pass Rate, FAIL 목록 등)
3. **PDF 다운로드**: 서식화된 리포트 파일 (회의/보고용)
4. **Excel 다운로드**: 상세 데이터 파일 (추가 분석용)
5. Confluence/Slack 등에 첨부하여 공유

## 4. 기능별 상세 가이드

### 4.1 프로젝트 목록 (홈 화면)

로그인 후 첫 화면입니다.

**구성 요소**:
- **전체 현황 카드**: 전체 프로젝트 수, 전체 TC 수, 진행률, Pass Rate
- **프로그레스 바**: PASS/FAIL/BLOCK/N/A/미수행 비율 색상 바
- **프로젝트별 테이블**: 프로젝트별 TC 수, 결과 분포, 진행률
- **프로젝트 카드**: 개별 프로젝트 카드 (클릭 시 상세 이동)

### 4.2 TC 관리

**툴바 기능 정리**:

| 버튼 | 기능 | 비고 |
|---|---|---|
| + 행 추가 | 새 TC 행 추가 | 그리드 하단에 빈 행 추가 |
| 선택 복사 | 선택한 행 복사 | 체크박스 선택 후 사용 |
| 선택 삭제 | 선택한 TC 삭제 | 소프트 삭제 (7일 후 영구삭제) |
| Undo / Redo | 실행 취소/재실행 | Ctrl+Z / Ctrl+Shift+Z |
| TC ID 자동채우기 | 빈 TC ID 자동 생성 | 프로젝트 약어 기반 |
| 일괄 변경 | 선택 행 필드 일괄 수정 | 우선순위, 카테고리 등 |
| Excel Import | 엑셀 TC 가져오기 | .xlsx, .xls (10MB 이하) |
| Excel Export | 엑셀 TC 내보내기 | 서식 적용된 엑셀 다운로드 |
| 변경이력 | TC 수정 내역 확인 | 필드별 변경 전/후 값, 변경자 |
| 바꾸기 | 찾기/바꾸기 | 선택 범위 텍스트 치환 |
| 검색 | TC 필터링 | TC ID, 카테고리, 내용 검색 |
| 저장 | 변경사항 서버 저장 | **편집 후 반드시 클릭** |

> **Warning**: 그리드 편집 후 **반드시 "저장" 버튼을 클릭**해야 변경사항이 서버에 반영됩니다. 페이지 이동 시 미저장 데이터는 유실됩니다.

### 4.3 테스트 수행

**런 상태 관리**:

| 상태 | 설명 | 전환 |
|---|---|---|
| **진행 중** | 테스트 수행 중 | 생성 시 기본 상태 |
| **완료** | 테스트 수행 완료 | "완료" 버튼 클릭 |

**첨부파일**:
- 지원 형식: 이미지(PNG, JPEG, GIF, BMP, WebP), 문서(PDF, DOC, XLSX, PPTX), ZIP, TXT, CSV, LOG
- 크기 제한: 50MB
- 이미지는 인라인 미리보기 제공
- 런 삭제 시 첨부파일도 자동 삭제

### 4.4 설정

**프로젝트 설정** (Admin/프로젝트 관리자):
- 프로젝트명, 설명 수정
- Jira Base URL 설정 (예: `https://yourteam.atlassian.net/browse/`)
- 공개/비공개 전환
- 프로젝트 삭제

**멤버 관리**:
- 사용자 추가 시 프로젝트 역할 부여: **Project Admin** / **Project Tester**
- 비공개 프로젝트는 멤버만 접근 가능
- 프로젝트 생성자는 제거 불가

## 5. 팀 운영 가이드

### 5.1 프로젝트 생성 기준

| 기준 | 설명 |
|---|---|
| **정기 배포** | 배포 버전별 프로젝트 생성 (예: `모바일 앱 v7.3 정기배포`) |
| **스프린트** | 스프린트 단위 프로젝트 (예: `결제 개선 Sprint 25`) |
| **핫픽스** | 긴급 수정 시 별도 프로젝트 (예: `결제 오류 핫픽스 20260316`) |
| **장기 리그레션** | 전체 리그레션 체크리스트 프로젝트 (지속 유지) |

### 5.2 TC 관리 기준

**TC 작성 원칙**:
1. **하나의 TC = 하나의 검증 포인트**: TC 하나에 여러 검증을 혼합하지 않음
2. **재현 가능한 절차**: 다른 사람이 읽고 동일하게 수행할 수 있도록 작성
3. **기대 결과 명확화**: "정상 동작" 대신 구체적인 결과 기술
4. **우선순위 기준 통일**:

| 우선순위 | 기준 |
|---|---|
| 매우 높음 | 핵심 업무 흐름, 결제/인증 등 크리티컬 기능 |
| 높음 | 주요 기능, 사용자가 자주 사용하는 기능 |
| 보통 | 일반 기능, UI/UX 검증 |
| 낮음 | 부가 기능, 엣지 케이스 |
| 매우 낮음 | 극히 드문 시나리오, 화면 텍스트 등 |

**Excel Import 시 주의사항**:
- 파일 크기: 10MB 이하
- 첫 번째 행 또는 헤더 행이 자동 감지됨
- 지원 헤더 예시: TC ID, Type, Category, Depth 1, Depth 2, Priority, Platform, Precondition, Test Steps, Expected Result 등
- 한글 헤더도 지원: 분류, 카테고리, 우선순위, 사전조건, 테스트 절차, 기대 결과 등

### 5.3 테스트 수행 기준

**FAIL 판정 기준**:

| 판정 | 조건 |
|---|---|
| FAIL | 기대 결과와 실제 결과가 불일치 |
| FAIL | 기능이 동작하지 않음 (에러, 크래시) |
| FAIL | 데이터 정합성 오류 |
| BLOCK | 선행 조건 버그로 해당 TC 수행 불가 |
| BLOCK | 테스트 환경 이슈로 수행 불가 |
| N/A | 요구사항 변경으로 해당 TC 불필요 |
| N/A | 테스트 범위에서 제외된 항목 |

**FAIL 시 필수 기록 항목**:
1. **Actual Result**: 실제 발생한 동작 (구체적으로)
2. **Issue Link**: Jira 이슈 키 (버그 티켓 생성 후)
3. **첨부파일**: 증빙 스크린샷 (가능한 경우)

### 5.4 리포트 공유 프로세스

| 시점 | 공유 대상 | 공유 방법 |
|---|---|---|
| 테스트 완료 후 | QA 리드, PM | PDF 리포트 + Confluence 페이지 |
| 리그레션 발견 시 | 개발팀 | 비교 화면 스크린샷 + Jira 이슈 |
| 배포 전 최종 | 전체 | Pass Rate 요약 + 잔여 FAIL 목록 |

### 5.5 라운드 관리

| 라운드 | 시점 | 목적 |
|---|---|---|
| **R1** | 개발 완료 직후 | 전체 기능 테스트, 버그 발견 |
| **R2** | 버그 수정 후 | 수정 확인 + 리그레션 테스트 |
| **R3** | 배포 전 | 최종 확인, 크리티컬 항목 재검증 |

- 라운드별로 테스트 런을 **복제(Clone)** 하여 생성하면 TC 구조가 유지됩니다.
- **비교 기능**으로 R1 → R2 → R3 간 개선/리그레션을 추적합니다.
- **대시보드 라운드별 비교 차트**로 추이를 시각화합니다.

### 6.4 백업

**백업 대상**:
1. 데이터베이스: `backend/tc_manager.db`
2. 첨부파일: `backend/uploads/` 디렉토리

**백업 명령**:
```bash
# DB 백업
cp backend/tc_manager.db backup/tc_manager_$(date +%Y%m%d).db

# 첨부파일 백업
cp -r backend/uploads backup/uploads_$(date +%Y%m%d)
```

**권장 백업 주기**: 일 1회 (자동화 권장)

### 6.5 트러블슈팅

## 7. FAQ

### Q1. YM TestCase에 접속할 수 없어요.
- 서버가 실행 중인지 확인합니다 (Backend: 8000, Frontend: 5173).
- 네트워크/VPN 연결을 확인합니다.
- 브라우저 캐시를 삭제하고 재접속합니다.

### Q2. 비밀번호를 잊었어요.
- Admin에게 연락하여 비밀번호 초기화를 요청합니다.
- 또는 새 계정을 생성합니다.

### Q3. TC를 실수로 삭제했어요.
- TC는 **소프트 삭제**됩니다. 7일 이내에 Admin에게 복원을 요청하세요.
- 복원 API: `POST /api/projects/{id}/testcases/{tc_id}/restore`

### Q4. 엑셀 Import 시 헤더를 인식하지 못해요.
- 지원하는 헤더 형식인지 확인합니다 (영문/한글 50여 가지 변형 지원).
- 헤더 행이 파일의 첫 번째 행 또는 두 번째 행에 있어야 합니다.
- 파일 형식이 .xlsx 또는 .xls인지 확인합니다.

### Q5. 다른 프로젝트의 TC를 검색하고 싶어요.
- 헤더의 **"TC 검색"** 입력란에 2자 이상 입력하면 전체 프로젝트에서 검색됩니다.
- 검색 결과를 클릭하면 해당 프로젝트의 TC로 이동합니다.

### Q6. 테스트 런을 삭제하면 첨부파일도 삭제되나요?
- 네, 런 삭제 시 관련 테스트 결과, 첨부파일 DB 레코드, 물리 파일이 **모두 영구 삭제**됩니다.
- 삭제 전 필요한 첨부파일은 미리 다운로드해두세요.

### Q7. 다크 모드를 사용하고 싶어요.
- 헤더의 🌙 아이콘을 클릭하면 다크/라이트 모드가 전환됩니다.
- 설정은 브라우저에 저장되어 재접속 시에도 유지됩니다.

### Q8. Jira 이슈와 연동하고 싶어요.
- 프로젝트 설정 → Jira Base URL에 Jira 주소를 입력합니다.
- TC 또는 테스트 결과의 Issue Link 필드에 이슈 키를 입력합니다 (예: PROJ-123).
- 클릭 시 해당 Jira 이슈 페이지로 이동합니다.
