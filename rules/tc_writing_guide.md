# TC 작성 가이드

## TC ID 규칙

- 형식: `TC-[모듈약어]-[번호]` (3자리 제로패딩)
- 예시: TC-PM-001, TC-AUTH-012, TC-TR-030

## TC 작성 원칙

### 기본 원칙

1. **하나의 TC = 하나의 핵심 검증 포인트** (기본)
2. **재현 가능한 절차**: 다른 사람이 읽고 동일하게 수행할 수 있도록 작성
3. **기대 결과 명확화**: "정상 동작" 대신 구체적 결과 기술
4. **번호 매기기**: 테스트 절차와 기대 결과에 단계별 번호 부여

### 예외 허용 (1TC 복수 검증)

다음 경우에는 하나의 TC에 여러 검증 포인트를 포함할 수 있습니다:

| 예외 | 예시 |
|---|---|
| **사용자 시나리오 전체 검증** | 회원가입 → 로그인 → 프로젝트 생성 → TC 등록 흐름 |
| **상태 전이 검증** | 테스트 런 생성 → 진행 중 → 완료 상태 전환 |
| **입력/출력 흐름 검증** | 엑셀 Import → 데이터 확인 → Export → 파일 비교 |
| **CRUD 전체 흐름** | TC 생성 → 조회 → 수정 → 삭제 → 복원 |

> **판단 기준**: 개별 단계로 쪼갰을 때 단독 검증 의미가 없으면, 하나의 TC로 묶는 것이 효율적입니다.

### TC 작성 예시

**좋은 예 (단일 검증)**:
```
TC ID: TC-AUTH-001
제목: 유효한 자격증명으로 로그인 성공
절차: 1. 로그인 페이지 접속  2. admin/admin123 입력  3. 로그인 클릭
기대: 1. 프로젝트 목록으로 이동  2. 헤더에 사용자명 표시
```

**좋은 예 (시나리오 흐름)**:
```
TC ID: TC-TR-050
제목: 테스트 런 전체 라이프사이클
절차: 1. 런 생성  2. 결과 3건 입력 (P/F/B)  3. 첨부파일 업로드  4. 런 완료  5. 리포트 확인
기대: 1. 런 생성됨  2. 결과 반영  3. 파일 첨부됨  4. 상태 "완료"  5. 리포트에 결과 포함
```

**나쁜 예 (검증 포인트 불명확)**:
```
TC ID: TC-PM-099
제목: 프로젝트 관리 확인
절차: 1. 프로젝트 관련 기능을 확인한다.
기대: 1. 정상 동작한다.
```

## 우선순위 기준

| 우선순위 | 기준 |
|---|---|
| 매우 높음 | 핵심 업무 흐름, 결제/인증 등 크리티컬 기능 |
| 높음 | 주요 기능, 자주 사용하는 기능 |
| 보통 | 일반 기능, UI/UX 검증 |
| 낮음 | 부가 기능, 엣지 케이스 |
| 매우 낮음 | 극히 드문 시나리오, 텍스트 확인 등 |

## FAIL 판정 기준

| 판정 | 조건 |
|---|---|
| **FAIL** | 기대 결과와 실제 결과 불일치, 기능 미동작, 데이터 오류 |
| **BLOCK** | 선행 조건 버그 또는 환경 이슈로 수행 불가 |
| **N/A** | 요구사항 변경으로 불필요, 테스트 범위 외 |

## FAIL 시 필수 기록

1. **Actual Result**: 실제 발생한 동작 (구체적으로)
2. **Issue Link**: Jira 이슈 키 (버그 티켓 생성 후)
3. **첨부파일**: 증빙 스크린샷 (가능한 경우)

## 라운드 관리

| 라운드 | 시점 | 목적 |
|---|---|---|
| **R1** | 개발 완료 직후 | 전체 기능 테스트, 버그 발견 |
| **R2** | 버그 수정 후 | 수정 확인 + 리그레션 테스트 |
| **R3** | 배포 전 | 최종 확인, 크리티컬 재검증 |

- 라운드별로 런을 **복제(Clone)** 하여 생성
- **비교 기능**으로 라운드 간 리그레션 추적
