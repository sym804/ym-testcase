# 버전 관리 규칙

## 버전 체계: `system.feature.fix.patch`

| 자리 | 의미 | 변경 기준 | 예시 |
|---|---|---|---|
| **system** | 시스템 구조 변경 | 대규모 리팩토링, 아키텍처 변경, 하위 호환 불가 | 0 → 1 (프로덕션 첫 배포) |
| **feature** | 기능 변경 | 신규 기능 추가, 기존 기능 대규모 수정 | 매뉴얼 페이지 추가 |
| **fix** | 중요 수정 | 사용자 영향이 큰 기능 오류, 보안 취약점, 빌드 실패, 데이터 정합성 | 빌드 에러 수정, 권한 검증 누락 수정 |
| **patch** | 경미한 수정 | UI 텍스트, 스타일, 사소한 동작, 운영 영향이 작은 보정 | 라벨 오타, 색상 조정, CORS 설정 변경 |

> **fix vs patch 판단 기준**: 해당 수정이 없으면 사용자가 기능을 못 쓰거나 보안 위험이 있으면 `fix`, 불편하지만 기능 자체는 동작하면 `patch`.

## 컴포넌트별 독립 버전

```
YM TestCase System  (전체 릴리즈 태그)
├── Frontend       (UI/UX 변경, 페이지 추가, 컴포넌트 수정)
├── Backend        (API 변경, 비즈니스 로직, 보안 수정)
└── Database       (테이블/컬럼 추가·변경·삭제, 마이그레이션)
```

- 각 컴포넌트는 **독립적으로** 버전이 올라갑니다.
- System Version은 전체 릴리즈 단위의 태그 역할이며, 컴포넌트 중 하나라도 변경되면 함께 올립니다.

## 버전 변경 규칙

| 변경 대상 | 올리는 버전 | 예시 |
|---|---|---|
| Frontend만 수정 | Frontend + System | FE 로그인 버그 → FE fix +1, System fix +1 |
| Backend만 수정 | Backend + System | BE 라벨 오타 수정 → BE patch +1, System patch +1 |
| DB 스키마 변경 | Database + Backend + System | 테이블 추가 → DB feature +1, BE 동반 수정 |
| 전체 변경 | 전체 | 대규모 기능 추가 |

## 기록 위치

- 릴리즈 노트: `Release_note.md`
- 이슈 목록: `Issue_list.xlsx`
- 각 릴리즈마다 컴포넌트별 버전 변경 테이블 포함
