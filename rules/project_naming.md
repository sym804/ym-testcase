# 프로젝트 네이밍 규칙

## 프로젝트명 형식

`[서비스명] [버전/스프린트] [환경] [유형]`

## 환경 식별자

| 식별자 | 의미 |
|---|---|
| `DEV` | 개발 환경 |
| `STG` | 스테이징 환경 |
| `PROD` | 운영 환경 |

## 유형 식별자

| 식별자 | 의미 |
|---|---|
| `REG` | 정기배포 테스트 |
| `SPRINT` | 스프린트 테스트 |
| `HOTFIX` | 긴급 수정 테스트 |
| `REGRESSION` | 장기 리그레션 |

## 예시

| 상황 | 프로젝트명 |
|---|---|
| 정기 배포 | `모바일 앱 v7.3 STG REG` |
| 스프린트 | `웹 결제 개선 Sprint25 STG SPRINT` |
| 핫픽스 | `결제 오류 20260316 PROD HOTFIX` |
| 장기 리그레션 | `모바일 앱 전체 STG REGRESSION` |
| 플랫폼별 | `iOS 앱 v7.3 STG REG` |

## 테스트 런 네이밍

`[서비스명] [날짜] [환경] [라운드]`

예시:
- `모바일 앱 20260316 STG R1`
- `결제 개선 20260320 STG R2`
- `결제 오류 20260316 PROD R1`

## YM TestCase 자체 프로젝트 예시

| 상황 | 프로젝트명 |
|---|---|
| v0.6.0 기능 테스트 | `YM TestCase v0.6 DEV REG` |
| v0.7.1 품질 게이트 | `YM TestCase v0.7.1 DEV REG` |
| 보안 점검 | `YM TestCase 보안점검 20260321 DEV HOTFIX` |

## 중복 방지

- 같은 서비스의 동일 버전이라도 **환경**이나 **유형**이 다르면 별도 프로젝트로 구분
- 날짜를 포함하여 시기별 구분
- 프로젝트 목록이 길어지면 완료된 프로젝트는 설명에 `[완료]` 표시 권장
